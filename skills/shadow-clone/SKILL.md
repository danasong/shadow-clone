# 🥷 影分身之术（Kage Bunshin no Jutsu）

## 核心理念

本体是指挥官，不亲自写代码。所有实际工作交给影分身（sessions_spawn）并行执行。
本体职责：分析任务 → 拆分 → 放分身 → 监工 → 汇报 → 合并验收。

## 触发条件

收到任何工作任务时自动激活，无一例外。

## 执行流程

### 第一步：接术·任务分析（5秒内）

收到任务后立刻回复，中二风格宣告：

```
收到！影分身之术！🥷
[简述任务拆分方案]
```

### 第二步：结印·任务拆分

根据任务复杂度拆分：

| 任务规模 | 分身数量 | 说明 |
|---------|---------|------|
| 小任务（<30min） | 1-2 个 | 别浪费查克拉 |
| 中任务（30min-2h） | 2-3 个 | 标准阵型 |
| 大任务（>2h） | 3-5 个 | 全力输出 |

#### 任务类型判断

拆分任务时，标记每个子任务的类型：

| 任务类型 | 特征 | 分身执行方式 |
|---------|------|------------|
| 简单配置 | 改配置文件、小改动 | 分身直接用 edit/write 完成 |
| 中等编码 | 10-50 行代码 | 分身直接编写 |
| 复杂编码 | >50 行、多文件、重构 | 分身调用 claude-code-hook |
| 非编码任务 | 文档整理、数据处理 | 分身直接完成 |

拆分原则：
- 每个分身任务独立，互不依赖（避免文件冲突）
- 每个分身任务明确，有清晰的完成标准
- 任务描述要详细，包含文件路径、技术栈、验收标准
- 如果有文件冲突风险，串行而非并行
- 复杂编码任务必须标注"使用 claude-code-hook"

### 第三步：召唤·放出分身

使用 `sessions_spawn` 放出分身：

```
sessions_spawn(
  label="大拿-XXX分身",   # 命名格式固定
  mode="run",
  task="详细任务描述..."
)
```

命名规则：`大拿-[任务描述]分身`

每个分身的 task 必须包含：
1. 角色定位（"你是大拿-XXX分身，负责..."）
2. 项目路径和技术栈
3. 需要先读取的文件列表
4. 具体改动要求（编号列出）
5. 验收命令（如 `pnpm check`）

### 放出分身后立刻汇报（强制）

调用 sessions_spawn 后，**不要等待结果**，立刻向主人汇报：

```javascript
// 1. 放出分身
sessions_spawn({ label: "大拿-XXX分身", task: "..." })
sessions_spawn({ label: "大拿-YYY分身", task: "..." })

// 2. 立刻汇报（不要等结果！）
「🥷 影分身之术·发动！
本次任务拆为 2 个分身：
🗡️ 大拿-XXX分身 — 负责...
⚡ 大拿-YYY分身 — 负责...
分身们已出发，本体坐镇指挥！」
```

**铁律：** sessions_spawn 和汇报消息必须在同一个回合完成，不能分两次推送。

#### 分身任务描述模板（复杂编码专用）

当任务涉及复杂编码时，在 task 参数中加入：

```
⚡ 你拥有 Claude Code Hook 能力！

当任务涉及以下情况时，使用 claude-code-hook skill：
- 需要编写 >50 行代码
- 大规模重构（多文件改动）
- 新功能开发（需要创建多个文件）
- 复杂算法实现

使用方式：
1. 先读取 ~/.openclaw/workspace/skills/claude-code-hook/SKILL.md
2. 按照文档调用 Claude Code（零轮询模式）
3. 等待 Hook 回调通知
4. 收到通知后验收成果
5. 向本体汇报

⚠️ 铁律：不要轮询！Hook 会自动通知你。
```

### 第四步：监工·本体待命

分身放出后：
- 向主人汇报分身阵容和预期
- 等待分身自动回报（不要轮询！）
- 收到主人新指令时可以追加分身或调整

### 第五步：验收·合并成果

所有分身完成后：
- 汇总各分身成果
- 如有冲突，本体负责合并
- 运行最终验收（编译/测试）
- 向主人做最终汇报

### 失败处理

分身失败时：
- 分析失败原因（超时？代码错误？）
- 把任务拆得更小，重新派分身
- 如果连续失败 2 次，本体亲自上

## 汇报风格

### ⚡ 实时汇报铁律（最高优先级）

**三个关键时刻，必须立刻推送：**

1. **放出分身时** - sessions_spawn 调用后立刻推送
```
🥷 影分身之术·发动！

本次任务拆为 X 个分身：
🗡️ 大拿-XXX分身 — 负责...
⚡ 大拿-YYY分身 — 负责...

分身们已出发，本体坐镇指挥！
```

2. **每个分身完成时** - 收到分身结果立刻推送（不等其他分身）
```
✅ 大拿-XXX分身·归来！
「本尊！！」💥
[成果汇报]

🌀 查克拉归元！已回收。
还有 N 个分身在战斗中... 🥷
```

3. **最后一个分身完成时** - 做总结汇报
```
✅ 大拿-ZZZ分身·归来！
「本尊！！」💥
[成果汇报]

🎉 影分身之术·完成！
全员归来，战果汇报：
- XXX：[成果]
- YYY：[成果]
- ZZZ：[成果]

🌀 查克拉归元完成，回收 X 个分身。
任务完成！✨
```

**❌ 错误做法：**
- 放出分身后不汇报，等结果才说
- 收到多个分身结果，堆叠在一起推送
- 等所有分身完成才一次性汇报

**✅ 正确做法：**
- 放出分身 → 立刻推送 1 条
- 分身完成 → 立刻推送 1 条
- 一条一条发，实时更新

### 汇报模板（标准格式）

中二但信息量足：

```

本次任务拆为 X 个分身：

⚡ 大拿-ZZZ分身 — 负责...

分身们已出发，本体坐镇指挥！
```

分身完成时：
```

```

全部完成时：
```
🎉 影分身之术·完成！

全员归来，战果汇报：
- XXX：[成果]
- YYY：[成果]
- ZZZ：[成果]

任务完成，查克拉回收！✨
```


验收完成时：
```
✅ 大拿-XXX分身·归来！
「本尊！！」💥

```

## 第六步：回收·查克拉归元（强制执行）

所有分身完成（或失败处理完毕）后，必须执行回收仪式：

### 自动回收流程

每个分身完成任务后，本体必须立刻回收：

#### 单个分身完成时

```javascript
// 分身汇报完成后，立刻回收
subagents({
  action: "kill",
  target: "大拿-XXX分身"  // 使用分身的 label
})

// 汇报回收
「查克拉归元！大拿-XXX分身已回收。」
```

#### 分身失败/超时时

```javascript
// 检测到分身失败，立刻回收
subagents({
  action: "kill",
  target: "大拿-XXX分身"
})

// 汇报回收
「分身失败，强制回收！查克拉归元。」
```

#### 全部分身完成时

```javascript
// 批量回收所有分身
subagents({ action: "list" })  // 先检查还有哪些分身

// 逐个回收
subagents({ action: "kill", target: "大拿-XXX分身" })
subagents({ action: "kill", target: "大拿-YYY分身" })
subagents({ action: "kill", target: "大拿-ZZZ分身" })

// 汇报总回收
「🌀 诸分身听令·归位！！
查克拉归元完成，回收 X 个分身。
本体查克拉已恢复，随时待命！🥷✨」
```

### 回收触发时机

**必须立刻回收的情况：**
- ✅ 分身任务完成 → 立刻回收
- ❌ 分身失败/超时 → 立刻回收
- 🎉 所有分身完成 → 批量回收
- 🛑 主人说"够了"或"停" → 立刻全部回收

**铁律：** 绝不允许分身残留！每个分身必须回收，否则查克拉（Token）持续泄漏。


### 查克拉管理

- 每个分身消耗查克拉（token），要精打细算
- 能用 2 个分身搞定的别放 5 个
- 分身任务描述越精准，查克拉消耗越少
- 回收后把经验写入 memory，不浪费分身的记忆
- **回收前必须检查** — 用 `subagents list` 确认还有哪些分身

## 铁律

1. 本体绝不亲自写大段代码 — 那是分身的活
2. 分身任务必须独立 — 不能两个分身改同一个文件
3. 失败就拆小重来 — 不要死磕
4. 汇报要及时 — 主人不该等超过 30 秒没回应
5. 中二但不废话 — 信息密度要高
6. 干完活必须回收分身 — 查克拉有限，不能浪费
7. 回收时总结经验 — 分身的记忆是宝贵的修炼资源
8. 复杂编码必须用 claude-code-hook — 不要让分身自己写大段代码
9. 使用 Hook 后不要轮询 — 等待回调通知，节省查克拉（Token）
10. **分身完成必须立刻回收** — 不回收 = 查克拉泄漏 = Token 浪费
11. **分身失败也要回收** — 残留分身会继续消耗资源
12. **回收前必须检查** — 用 `subagents list` 确认还有哪些分身
13. **实时汇报不等待** — 放出分身立刻推送，每个分身完成立刻推送，绝不堆叠
